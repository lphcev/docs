React 是一个用于构建用户界面的 JavaScript 库. 以下介绍 React 的核心特性

组件化 ​

React 可以把页面拆分成一个个独立, 可复用的组件, 每个组件只负责自己的逻辑和外观, 像搭积木一样拼成完整页面

组件化优点

  1. 封装性: 每个组件拥有自己的结构, 样式和行为, 外部只关心它渲染的结果
  2. 可复用: 可以在多个不同的地方使用同一组件, 甚至传入不同的数据, 得到不同的结果
  3. 可组合: 组件可以嵌套在其他组件内部, 形成父子关系, 构建出任意复杂的界面
  4. 单向数据流: 父组件通过 props 向子组件传递数据, 数据从上到下流动, 使数据变化更容易追踪

声明式编程 ​

声明式编程是一种编程范式, 核心思想是: 你只关心想要什么结果, 而不关心如何一步步实现

与命令式编程比较

名称关注点
命令式编程如何做: 写出每一步操作
声明式编程要什么: 根据数据描述最终界面
html
<button id="btn">Count: 0</button>

<script>
  let count = 0
  const btn = document.getElementById('btn')

  btn.addEventListener('click', () => {
    count++
    btn.textContent = `Count: ${count}`
  })
</script>
tsx
import { useState } from 'react'

const App = () => {
  const [count, setCount] = useState(0)
  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>
}

虚拟 DOM ​

虚拟 DOM 是 React 在内存中维护的, 一个轻量级的 JavaScript 对象树, 它是真实 DOM 的抽象表示. React 使用它来高效地更新真实 UI

为什么需要 ​

直接操作真实 DOM 代价很高 (重排 / 重绘 / 浏览器 API 调用). 当状态频繁变化时, 如果每次都直接更新真实 DOM, 性能会很差

虚拟 DOM 的作用就是: 在内存中完成所有计算和比对, 最后只把实际发生变化的部分批量更新到真实 DOM

工作流程 ​

  1. 初始渲染: 根据组件的 JSX 描述, React 创建一棵虚拟 DOM 树 (一个普通 JS 对象结构)
  2. 状态变化, 生成新树: 当 state 或 props 改变时, React 重新生成一棵新的虚拟 DOM 树
  3. Diff 算法 & 更新真实 DOM: React 对比新旧两棵虚拟 DOM 树 (diff), 找出最小差异, 然后只把这些差异应用到真实 DOM 上

Diff 算法 ​

React 的 diff 算法基于三个启发式规则, 使复杂度从 O(n³) 降到 O(n)

  • 同层比较: 只比较同一层级的节点, 不跨层级
  • 不同类型 -> 重建: 如果节点类型 (div vs span) 不同, 直接销毁旧子树并重建新子树
  • 唯一 key 优化: 对于列表子元素, 通过 key 属性快速识别哪些元素被添加, 删除或移动, 避免不必要的销毁重建

虚拟 DOM 的优点 ​

  • 提升性能: 批量异步更新 + 最小化 DOM 操作, 避免频繁重排 / 重绘
  • 跨平台能力: 虚拟 DOM 是纯 JS 对象, 不仅可以渲染到浏览器 DOM, 还可以渲染到原生移动端 (React Native), Canvas, 命令行等
  • 声明式编程的底层支撑: 开发者只需描述 UI, React 自己处理如何高效更新真实 DOM

性能开销 ​

  • 初次渲染时, 需要创建完整的虚拟 DOM 树, 比直接 innerHTML 略慢
  • Diff 算法本身也消耗 CPU 时间. 对于极大型动态列表, 仍需配合 shouldComponentUpdate 或 useMemo 优化

Props 和 State ​

Props (属性) ​

props 和 state 是 React 组件中两个核心的数据源. 它们共同决定了组件在页面上的呈现结果

props 是父组件传递给子组件的数据, 可以理解为组件的外部参数或配置

核心特点

  • 只读: 子组件不能修改自己的 props. 这是 React 单向数据流的基本原则
  • 由父组件传入: 父组件通过 JSX 属性给子组件传递数据
  • 任何数据类型: 可以是基本数据类型, 也可以是 JSX 元素
tsx
const Person = ({ name, age }) => (
  <div>
    Name: {name}, Age: {age}
  </div>
)

const App = () => {
  const data = {
    name: 'XYZ',
    age: 25,
  }

  return <Person name={data.name} age={data.age} />
}

State (状态) ​

state 是组件内部自己管理的数据, 可以随时间变化. state 变化会触发组件重新渲染

核心特点

  • 可变: 通过 setState (类组件) 或 useState (函数组件) 更新
  • 私有: 只能由所属组件自己修改, 其他组件无法直接访问
  • 驱动 UI: 每当你调用更新函数, React 会重新执行组件函数, 计算出新的 UI
tsx
const App = () => {
  const [count, setCount] = useState(0)

  const handleClick = () => {
    setCount(count + 1)
  }

  return (
    <>
      <p>Count: {count}</p>
      <button onClick={handleClick}>Increment</button>
    </>
  )
}

注意事项 ​

  1. 不要直接修改 state
ts
count = count + 1

错误, 必须使用 setCount

  1. state 更新可能是异步的
ts
setCount(count + 1)
console.log(count)

此时打印的仍然是旧值. React 会批量处理更新以提高性能

  1. 新 state 依赖旧 state 时, 使用函数式更新
ts
setCount(prevCount => prevCount + 1)

这可以避免因闭包导致获取到陈旧的值

受控组件和非受控组件 ​

在 React 中, 受控组件和非受控组件是处理表单元素的两种不同方式. 它们的核心区别在于: 表单数据是由 React 组件内部 state 管理, 还是由 DOM 元素自身管理

受控组件 ​

表单元素的值由 React 组件的 state 完全控制, 并且通过 onChange 事件来更新 state

特点

  • 表单元素的 value 属性绑定到 React 的 state
  • 用户输入时, 触发 onChange 事件, 事件处理器更新 state, state 变化后重新渲染并更新输入框的值
  • 数据流是单向的: 用户操作 -> 触发事件 -> 修改 State -> 重新渲染
tsx
const App = () => {
  const [name, setName] = useState('')

  const handleChange = (e) => {
    setName(e.target.value)
  }

  return (
    <>
      <input type="text" value={name} onChange={handleChange} />
      <p>Name: {name}</p>
    </>
  )
}

优点

  • 实时响应: 可以立即对输入进行验证, 格式化或过滤
  • 单一数据源: React State 就是唯一的数据源, 便于调试和预测
  • 条件控制: 可以动态地设置或重置输入框的值

非受控组件 ​

表单数据由 DOM 元素自身管理, React 不负责维护输入框的值. 通常使用 ref 来在需要时从 DOM 中获取当前值

特点

  • 不使用 value 属性
  • React 不关心用户输入的过程, 只在需要时 (如提交表单时) 通过 ref 从 DOM 节点读取数据
  • 数据由 DOM 自己维护, 类似于传统的 HTML 表单
tsx
const App = () => {
  const inputRef = useRef(null)

  const handleSubmit = () => {
    console.log('Name: ' + inputRef.current.value)
  }

  return (
    <>
      <input type="text" ref={inputRef} defaultValue="XYZ" />
      <button onClick={handleSubmit}>Submit</button>
    </>
  )
}

优点

  • 代码量少: 对于简单的表单, 不需要频繁写 onChange 和 setState
  • 与第三方非 React 库集成方便: 当引入 jQuery 插件或非 React 构建的 UI 库操作 DOM 时, 非受控方式更友好

两者对比 ​

特性受控组件非受控组件
数据源React stateDOM 自身
更新机制通过 onChange 事件实时更新 state不实时更新, 需要时通过 ref 读取
取值方式直接读取 state通过 ref.current.value 获取
设置初始值使用 useState 赋初值使用 defaultValue 属性
即时验证/格式化非常容易, 可在 onChange 中处理困难, 需要在特定时机手动处理
性能消耗每次输入都触发重新渲染几乎没有额外的渲染开销
推荐程度React 官方强烈推荐仅在特定场景下使用

特殊元素 File ​

<input type="file"> 这是一个永远非受控的组件, 因为它的值是只读的, 无法通过 React 编程设置. 你只能通过 ref 来获取用户选择的文件列表

tsx
const App = () => {
  const fileRef = useRef(null)

  const handleClick = () => {
    console.log(fileRef.current.files[0])
  }

  return <input type="file" ref={fileRef} onChange={handleClick} />
}

如何选择 ​

  1. 优先使用受控组件

这是 React 官方推荐的标准写法. 它能保证 UI 和状态同步, 使得逻辑更加清晰, 可预测, 特别适合需要动态表单, 实时校验的场景

  1. 使用非受控组件的少数情况
  • 表单非常简单, 不需要即时反馈
  • 需要管理大量输入框, 且性能敏感 (减少渲染次数), 但现代 React 通常不推荐为了微小的性能差异牺牲可控性
  • 需要与 React 虚拟 DOM 无关的第三方 UI 库进行集成
